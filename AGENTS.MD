# JM Boom Project Notes

本文档记录 `feat/docker` 分支放弃 Tauri、迁移为 React Web 前端与 Axum 后端后的当前架构约束、已确认问题和后续工作顺序。内容应保持短而准；问题修复后及时删除或更新对应条目，不记录过程流水账。

## 当前架构与约定

- 项目已完全放弃 Tauri，当前结构为 `web/` React 前端与 `server/` Rust/Axum 后端。
- 前端使用 React 19、Vite、TanStack Router/Query、Zustand、shadcn/ui。
- 后端负责 JM API 请求、端点探测、登录态、阅读图片解扰、缓存和下载任务。
- 前端依赖、类型检查、构建、lint 和格式验证统一使用 Bun，不使用 pnpm；不要用 npm 改写依赖锁。
- 不要手动编辑 `web/src/routeTree.gen.ts`，路由树由 TanStack Router 插件生成。
- 读取或写入中文文件时显式使用 UTF-8。
- 文件编辑使用 `apply_patch`，保留用户已有的无关改动。
- Docker 镜像、静态资源托管、基础镜像版本、构建上下文等容器化问题属于后续阶段，当前阶段不作为功能对齐阻断项。
- 产品界面采用移动端优先、桌面端完整适配的响应式方案；主导航在所有尺寸下都使用底部胶囊形态，不恢复桌面左侧导航。
- 主导航固定为书架、探索、收藏、下载、设置。书架使用 `ShellIcon`；收藏、下载、设置沿用现有图标。
- 探索负责推荐、每周、排行、搜索和首页分区列表；漫画详情与阅读器隐藏底部导航。
- 收藏只保存到当前浏览器，不依赖登录或上游云收藏。

## 已保留的非 Tauri 功能

- 首页、首页分区列表、每周推荐、排行榜和搜索。
- 漫画详情、章节、相关推荐、评论、收藏。
- 单页、双页、条漫、键盘/点击翻页和自动阅读。
- 本地阅读历史。
- 服务端上游接口发现、测速、自动选择和故障切换。
- 服务端阅读缓存和批量章节缓存任务。

以下能力属于 Tauri/桌面运行时能力，放弃是预期行为，不作为缺失：应用内更新安装、系统目录打开、桌面日志目录、Tauri IPC、桌面窗口/托盘和本地文件 URL 转换。

## 当前优先问题

### P0：会话和安全边界不适合 Web 服务

- `AppState` 只有一个全局用户会话，`JmClient` 也只有一个全局 JWT。不同浏览器和用户会共享账号、收藏和签到状态，一个用户退出会清除所有人的登录态。
- API 没有客户端会话标识或鉴权中间件，CORS 当前允许任意 Origin。即使按单用户服务设计，外部网页也可能跨域调用本机 API 操作当前账号。
- 需要改为每客户端独立的服务端会话或安全 Cookie；上游 JWT 必须按会话隔离。生产环境默认同源，开发环境只允许明确的前端 Origin。
- 服务重启会丢失会话，`master` 中的自动登录能力没有等价替代。Web 架构不应继续保存明文密码式自动登录，优先设计可持久化、可撤销的服务端会话。

### P0：退出登录前后端协议不一致

- `DELETE /api/auth/session` 返回空响应，但 `web/src/lib/api/client.ts` 对所有成功响应强制执行 `response.json()`。
- 后端会话清除后前端仍会抛错，Zustand 用户状态不会清空，页面显示与服务端实际状态不一致。
- HTTP 客户端必须正确处理 `204`、空响应和非 JSON 成功响应，或让退出接口返回统一 JSON。

### P1：下载任务状态机不正确

- 暂停任务时 worker 留在轮询循环中；恢复任务会清零总页数、完成页数和速度，但旧 worker 会从原位置继续，造成进度跳变、完成数超过总数或状态长期停留在 `queued`。
- 服务重启后，原 `running` 任务改为 `queued`，但旧进度没有统一重置，重新遍历章节时可能重复累计。
- 删除任务目前只删除内存和数据库记录，不删除关联图片；前端却提示“删除任务和已保存文件”。必须统一产品语义。
- 每个任务都会立即 `tokio::spawn`，没有全局下载并发限制。需要显式队列和 Semaphore，避免大量任务同时占满网络、CPU 和 SQLite。
- 下载进度每页写一次完整 JSON 到 SQLite，缓存命中还会额外更新访问时间；应降低写放大并避免持有任务写锁跨数据库 `await`。

### P1：离线缓存语义未完成

- 当前“下载”实际是预热阅读缓存，不是 `master` 中独立、可管理的下载文件。
- 下载任务没有持久化章节 manifest。阅读器打开章节时仍需访问上游取得图片列表，因此上游不可用时，即使页面已缓存，也不保证可以离线阅读。
- 需要决定产品语义：若继续称为“离线缓存”，必须持久化章节 manifest、页列表和任务到缓存文件的归属关系；删除任务时才能准确删除对应文件。

### P1：缓存无容量管理且图片被无条件有损重编码

- `cache_index` 目前只写入和更新时间，没有容量限制、统计、LRU 淘汰、清理或孤儿文件修复逻辑，磁盘会持续增长。
- 所有非 GIF 图片都会解码为 RGB 并重新编码为质量 75 的 WebP，即使图片不需要解扰。这会增加 CPU、造成二次有损压缩，并丢失 PNG/WebP alpha 通道。
- 不需要解扰的图片应优先原样缓存；只有需要解扰的图片才进行解码和重新编码，并保留必要的颜色/透明度信息。
- 页面响应使用一年 `immutable`，但公开 URL 不包含缓存或解扰算法版本。算法更新后浏览器仍可能长期使用旧图；版本必须进入 URL/ETag，或取消 immutable。
- 需要恢复服务端缓存统计、容量配置和清理 API；“打开缓存目录”属于 Tauri 能力，可继续放弃。

### P1：阅读器预取兼容层没有真正预取图片

- `web/src/lib/api/reader.ts` 的 `getComicReadPage()` 只返回路径和固定 `800x1200` 假尺寸，没有发起图片请求。
- TanStack Query 的相邻页、下一章预取大多只缓存 URL，实际图片仍由 `<img>` 首次渲染时下载。
- 应直接以图片 URL 驱动浏览器预加载，或在预取函数中真实 fetch/blob；删除固定尺寸、`isCached`、`requestOrigin` 等不再有意义的 Tauri 兼容字段。

### P2：搜索功能存在回退缺失和遗留协议

- 后端保留上游 `redirect_aid`，前端声明但不处理。部分搜索结果只有重定向漫画 ID 时会错误显示为空。
- `search.ts` 仍构造旧插件协议的 `source/scheme/data/extern`，当前页面实际只使用 `items` 和 `paging`。
- 应在后端完成 redirect fallback，并让 HTTP API 返回前端真正需要的简洁 DTO。
- 当前搜索结果把 `updatedAt` 写成请求时的 `Date.now()`，不应伪造更新时间；没有真实值时使用 `null`。

### P2：上游响应反序列化健壮性较 master 退化

- 漫画详情中的 `author/tags/actors/works` 直接使用 `Vec<String>`，收藏状态直接使用 `bool`。上游若返回标量、数字字符串或 `0/1`，整个接口可能反序列化失败。
- `master` 已存在数组/标量、字符串/数字/布尔混合字段的宽松解析逻辑；迁移时应复用或集中实现，不能在 `auth.rs`、`comics.rs`、`home.rs`、`models.rs` 中复制多套近似 helper。
- 为真实上游响应样本增加反序列化回归测试。

### P2：长时间运行的内存和并发管理

- 页面物化锁、章节请求锁和图片 Host 请求锁使用全局 `HashMap<String, Arc<Mutex<_>>>`，完成后不移除。
- 章节缓存只在再次访问同一个过期 key 时删除，长期运行会按访问过的章节和页面持续增长。
- 应使用有容量/TTL 清理的缓存，或在任务完成后安全移除单次飞行锁。
- 阅读器后台预热、下载任务和图片解码都使用无界 `tokio::spawn`/`spawn_blocking`，需要共享并发预算。

## 设置页尚未对齐的非 Tauri 能力

- 缓存容量、缓存统计和清理：应迁移到服务端，不能因放弃 Tauri 一并删除。
- 网络代理：代理作用对象现在是服务端 HTTP Client，应改为服务端环境变量或受保护的管理配置，而不是浏览器本地设置。
- 账号会话管理：需要展示当前服务端会话、退出其他会话或撤销会话，不应照搬桌面端保存密码方案。
- 版本展示、检查更新和安装更新属于不同概念：安装更新可放弃；显示 Web/Server 构建版本仍有诊断价值，但优先级较低。
